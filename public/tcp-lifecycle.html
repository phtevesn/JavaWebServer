TCP Connection Lifecycle

Understanding TCP connection establishment and teardown is crucial for understanding why keep-alive matters for performance.

The 3-Way Handshake (Connection Establishment)

Before any data can be sent, TCP requires a 3-way handshake:

Client Server | | |-------- SYN ---------> | (1) Client: "I want to connect" | | |<---- SYN-ACK --------- | (2) Server: "OK, I acknowledge" | | |-------- ACK ---------> | (3) Client: "I acknowledge your acknowledgment" | | | Connection ESTABLISHED |

Cost: 1.5 round-trip times (RTTs) before any data flows!

The 4-Way Handshake (Connection Termination)

Closing a connection requires 4 packets:

Client Server | | |-------- FIN ---------> | (1) Client: "I'm done sending" | | |<------- ACK ---------- | (2) Server: "I acknowledge" | | |<------- FIN ---------- | (3) Server: "I'm done too" | | |-------- ACK ---------> | (4) Client: "I acknowledge" | | | Connection CLOSED |

Cost: 2 RTTs to close gracefully.

TIME_WAIT State: The Port Exhaustion Problem

After closing a connection, the client-side port enters TIME_WAIT state for typically 60 seconds:

Client connects to server: Local port: 52341 Remote: server:8080 Connection closes Port 52341 enters TIME_WAIT for 60 seconds Port 52341 CANNOT be reused during this time Why? To ensure delayed packets don't interfere with new connections
‚ö†Ô∏è The Problem: With 100,000 requests and no keep-alive, you need 100,000 different local ports. But there are only ~28,000 ephemeral ports available! Result: "Cannot assign requested address" error.

Without Keep-Alive (100 Requests)

Request #1: TCP handshake (3 packets) HTTP request + response TCP teardown (4 packets) Port enters TIME_WAIT Request #2: TCP handshake (3 packets, NEW PORT) HTTP request + response TCP teardown (4 packets) Port enters TIME_WAIT ... repeat 98 more times ... Total: 100 handshakes, 100 ports in TIME_WAIT

With Keep-Alive (100 Requests)

Connection established: TCP handshake (3 packets, ONE PORT) Request #1: HTTP request + response Request #2: HTTP request + response Request #3: HTTP request + response ... (all on same connection) ... Request #100: HTTP request + response Connection closes: TCP teardown (4 packets) ONE port enters TIME_WAIT Total: 1 handshake, 1 port in TIME_WAIT

Savings: 99 fewer handshakes, 99 fewer ports!

üí° Key Insight: Keep-alive isn't just faster - it's necessary at scale. Without it, you literally run out of ports!

Ephemeral Port Range

On Linux, check your ephemeral port range:

$ cat /proc/sys/net/ipv4/ip_local_port_range 32768 60999 # This system has 28,232 available ports

With TIME_WAIT lasting 60 seconds, you can exhaust ports very quickly without keep-alive!
