Web Server Security Basics

Even a simple web server needs security measures to prevent attacks. This guide covers the essential security principles implemented in our server.

1. Directory Traversal Prevention

üö® Critical Vulnerability: Never trust paths from URLs!

The Attack

GET /../../../etc/passwd HTTP/1.1

Without protection, this could allow attackers to read any file on the system!

The Defense

// Check for ".." in path if (path.contains("..")) { sendError(out, 403, "Forbidden"); return; } // Alternative: Use canonical paths Path requestedPath = Paths.get(PUBLIC_DIR, path); Path publicPath = Paths.get(PUBLIC_DIR).toRealPath(); if (!requestedPath.toRealPath().startsWith(publicPath)) { sendError(out, 403, "Forbidden"); return; }

2. Resource Exhaustion (DoS) Prevention

Payload Size Limits

int size = Integer.parseInt(sizeParam); // ‚ùå WITHOUT LIMIT: Client requests /echo/2147483647 // Server tries to allocate 2GB ‚Üí OutOfMemoryError! // ‚úÖ WITH LIMIT: Enforce reasonable maximum if (size < 0 || size > 100_000_000) { sendError(out, 400, "Size out of range (0-100MB)"); return; }

Connection Timeouts

// Prevent clients from holding connections forever socket.setSoTimeout(keepAliveTimeout * 1000); // Without timeout: Malicious client connects and never sends data // ‚Üí Server thread blocked forever

3. Input Validation

Input Validation Why
Request line Check for null, empty, valid parts Prevent NullPointerException
Path Check for "..", validate format Prevent directory traversal
HTTP method Whitelist allowed methods (GET) Reject unsupported operations
Size parameter Parse as int, check range Prevent DoS and errors

4. Safe Error Messages

// ‚ùå BAD: Expose internal details catch (Exception e) { sendError(out, 500, "Error: " + e.getMessage() + " at " + e.getStackTrace()[0]); } // Reveals code structure to attackers! // ‚úÖ GOOD: Generic error message catch (Exception e) { System.err.println("Internal error: " + e.getMessage()); sendError(out, 500, "Internal Server Error"); } // Log details server-side, don't expose to client

5. Resource Cleanup

// Always close sockets to prevent resource leaks try (Socket clientSocket = serverSocket.accept()) { handleConnection(clientSocket); } // Auto-closed even if exception occurs // For virtual threads, explicitly close in finally: finally { try { clientSocket.close(); } catch (IOException e) { // Log but don't crash } }

6. Header Injection Prevention

‚ö†Ô∏è Warning: If you ever use user input in headers, sanitize it!
// ‚ùå DANGEROUS: User input in header String userValue = getUserInput(); writer.print("X-Custom: " + userValue + "\r\n"); // Attack: userValue = "bad\r\nX-Evil: injected" // Result: Multiple headers injected! // ‚úÖ SAFE: Strip CRLF characters String sanitized = userValue.replace("\r", "") .replace("\n", ""); writer.print("X-Custom: " + sanitized + "\r\n");

7. Method Validation

// Only allow specific HTTP methods if (!method.equals("GET")) { sendError(out, 405, "Method Not Allowed"); return; } // Principle: Whitelist allowed operations, // reject everything else by default

Security Checklist

  • ‚úÖ Validate all input (paths, parameters, headers)
  • ‚úÖ Check for directory traversal (".." in paths)
  • ‚úÖ Enforce resource limits (size, timeout)
  • ‚úÖ Use try-with-resources for automatic cleanup
  • ‚úÖ Generic error messages (don't expose internals)
  • ‚úÖ Whitelist HTTP methods (reject unknown methods)
  • ‚úÖ Set socket timeouts (prevent resource holding)
  • ‚úÖ Log security events server-side
üí° Defense in Depth: Multiple layers of security work together. Even if one check fails, others catch the problem. Never rely on a single security measure!

Testing Security
